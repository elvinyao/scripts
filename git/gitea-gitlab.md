GiteaリポジトリのGitLabへの移行・分割および管理に関するご提案書I. エグゼクティブサマリー
概要: 現在、プロジェクトのバージョン管理はGitea上で行われていますが、管理されていないバイナリファイルディレクトリ (bot-release) を含むため、リポジトリサイズが非常に大きくなっています。バージョン管理は主にタグに依存しており、効果的なブランチ戦略が欠けています。本提案の目的は、GiteaリポジトリをGitLabに移行し、2つの独立したプロジェクトに分割することです。まず、ローカル環境でGiteaリポジトリの履歴をクリーンアップし、bot-releaseディレクトリとその履歴を完全に削除してリポジトリサイズを削減します。この際、既存のすべてのタグは保持します。次に、クリーンアップされた履歴をさらに分割します。Apex関連ディレクトリ (apex-apl, apex-db, apex-api) とその関連履歴およびタグを新しいGitLabプロジェクトapex-projectに移行します。Bot関連のソースコードとテストディレクトリ (bot-apl/bot-src, bot-unittest) とその関連履歴およびタグは、新しいGitLabプロジェクトbot-src-archiveにアーカイブとして移行します。最後に、apex-projectに対して、最新のGitワークフローと管理規範を確立します。
核心的なご提案:

クリーンアップ: Giteaリポジトリのローカルミラークローン上で、git filter-repo --invert-paths --path bot-release/ コマンドを使用し、bot-release ディレクトリの履歴を完全に削除します。この際、--tags オプション（デフォルトで有効）により、すべてのタグが保持されるようにします。
分割 (Apex): クリーンアップ済みリポジトリのコピー上で、再度 git filter-repo --path apex-apl/ --path apex-db/ --path apex-api/ コマンドを使用し、Apex関連ファイルとその履歴、関連タグのみを抽出します。この結果をGitLab上に新規作成した apex-project リポジトリにプッシュします。
分割 (Bot): クリーンアップ済みリポジトリの別のコピー上で、git filter-repo --path bot-apl/bot-src/ --path bot-unittest/ コマンドを使用し、Botソースコードとテスト関連ファイルとその履歴、関連タグのみを抽出します。この結果をGitLab上に新規作成した bot-src-archive リポジトリにプッシュします。
管理 (apex-project): apex-project リポジトリ（主にSQLコードを含む）に対して、GitHub Flow または Trunk-Based Development (TBD) を新しいブランチモデルとして採用し、GitLabマージリクエストに基づいたコードレビュープロセスを導入します。SQLコードのバージョン管理に関するベストプラクティスに従います。


主要なメリット:

リポジトリの分割により、コンポーネント間の分離を実現します。
apex-project のリポジトリサイズを大幅に削減し、パフォーマンスを向上させます。
apex-project のための明確で効率的なSQL開発ワークフローを確立します。
GitLabのコードレビュー機能を通じてSQLコードの品質を向上させます。
Botの履歴コードを安全にアーカイブします。


実行計画概要: 移行プロセスは以下のフェーズで構成されます：現状分析 -> ローカルクローンとクリーンアップ -> Apexコード分割 -> Botコード分割 -> GitLabへのプッシュ -> 結果検証 -> apex-project の新ワークフロー導入。
II. Giteaリポジトリの現状分析

A. リポジトリ構造と内容:

現在のGiteaリポジトリには、主に5つのフォルダが含まれています：apex-apl, apex-db, apex-api, bot-apl, bot-unittest。
bot-apl フォルダ内には bot-src (ソースコード) と bot-release (バイナリファイル) があります。
bot-release ディレクトリが大量のバイナリファイルを格納しており、リポジトリサイズ肥大化の主因であることを確認しました。

Giteaリポジトリ (現状)
├── apex-apl/
├── apex-db/
├── apex-api/
├── bot-apl/
│   ├── bot-src/      <-- Botソースコード
│   └── bot-release/  <-- ★肥大化の原因 (バイナリ)
└── bot-unittest/     <-- Botテストコード



B. ボリュームと履歴評価:

操作: Giteaリポジトリのローカルクローン上で git filter-repo --analyze コマンドを実行し、リポジトリ履歴を分析します。
分析: 生成されたレポート（.git/filter-repo/analysis/ 内）を確認します。特に *-sizes.txt ファイルは、bot-release/ が占める容量を定量的に示し、削除の必要性を裏付けます。また、他の隠れた巨大ファイルがないかも確認します。
評価結果: bot-release が主要な問題源であることを確認します。分析レポートにより、他の潜在的な巨大ファイルが見つかった場合は、それらもクリーンアップ対象とするか、apex-project や bot-src-archive でGit LFSの導入を検討する必要があります。



C. ブランチとタグの実践:

現状：バージョン管理は主にタグに依存しており、ブランチ戦略は限定的です (master は長期間更新されず、ver1.0 ブランチのみが存在)。
影響：並行開発が困難で、履歴追跡が複雑化し、将来的なマージ競合のリスクが高まります。タグが巨大ファイルを含む履歴を指しているため、履歴のクリーンアップが不可欠です。


III. 移行、履歴クリーンアップ、分割戦略

A. 移行方法: 手動での移行および分割プロセスを採用します。GitLabのGiteaインポート機能では、インポート中の履歴フィルタリングやリポジトリ分割はサポートされていません。


B. 履歴書き換えツール: git filter-repo の使用を推奨します。これは現代的で高速、安全であり、指定したパスに基づいて履歴を正確にフィルタリングし、デフォルトですべてのブランチとタグを処理します。


C. 推奨される移行・分割プロセス:
プロセス全体の流れ図:
代码段graph LR
    A[Giteaリポジトリ] --> B(ローカルにミラークローン);
    B --> C{git filter-repo\n(bot-release削除)};
    C --> D[クリーンアップ済み\nローカルリポジトリ\n(タグ保持)];
    D --> E(コピー1作成);
    E --> F{git filter-repo\n(Apex抽出)};
    F --> G([GitLab: apex-project]);
    D --> H(コピー2作成);
    H --> I{git filter-repo\n(Bot抽出)};
    I --> J([GitLab: bot-src-archive]);



フェーズ1：ローカルでのクリーンアップ

クローン: Giteaリポジトリの新しい裸のミラークローンを作成します: git clone --mirror <gitea_repo_url> migration_repo_cleaned。作成したディレクトリに移動します: cd migration_repo_cleaned。必ず新しいクローン上で操作してください。
git filter-repoインストール: Python 3 と git-filter-repo がインストールされていることを確認します。
分析 (任意): git filter-repo --analyze を実行して影響を確認します。
クリーンアップ実行: git filter-repo --invert-paths --path bot-release/ を実行します。これにより bot-release/ ディレクトリの全履歴が削除され、すべてのブランチとタグが新しいクリーンなコミットを指すように自動的に更新されます。
クリーンアップ検証: 履歴 (git log --all --stat) とタグ (git tag -l) を確認し、bot-release が削除され、すべての元のタグが存在すること（ただし新しいコミットを指している）を確認します。リポジトリサイズ (git count-objects -vH) が減少したことを確認します。
ガベージコレクション: git filter-repo は通常、自動で gc を実行します。必要であれば手動で git reflog expire --expire=now --all && git gc --prune=now --aggressive を実行します。



フェーズ2：apex-project の分割とプッシュ

コピー作成: クリーンアップ済みの裸リポジトリ migration_repo_cleaned を新しいディレクトリにコピーします (例: cp -r migration_repo_cleaned apex_project_filtered)。新しいディレクトリに移動します: cd apex_project_filtered。次の分割のために、クリーンアップ済みリポジトリの完全なコピーを保持するため、必ずコピー上で操作してください。
Apexフィルタリング実行: git filter-repo --path apex-apl/ --path apex-db/ --path apex-api/ を実行します。これにより、指定されたApexディレクトリに関連する履歴とファイルのみが保持されます。git filter-repo は、これらの残りのコミットを指すすべてのタグを自動的に更新します。
Apexフィルタリング検証: 履歴、ファイル、タグを確認し、Apex関連のコンテンツのみが含まれ、関連するタグが保持されていることを確認します。
GitLabへのプッシュ:

GitLabに apex-project という名前の空のプロジェクトを作成します。
リモートリポジトリを追加します: git remote add gitlab <apex_project_gitlab_url>。
ミラープッシュを実行します: git push --mirror gitlab。





フェーズ3：bot-src-archive の分割とプッシュ

別のコピー作成: クリーンアップ済みの裸リポジトリ migration_repo_cleaned をさらに別の新しいディレクトリにコピーします (例: cp -r migration_repo_cleaned bot_src_archive_filtered)。新しいディレクトリに移動します: cd bot_src_archive_filtered。
Botフィルタリング実行: git filter-repo --path bot-apl/bot-src/ --path bot-unittest/ を実行します。これにより、指定されたBotソースコードとテストディレクトリに関連する履歴とファイルのみが保持され、関連するタグが更新されます。
Botフィルタリング検証: 履歴、ファイル、タグを確認し、Bot関連のコンテンツのみが含まれ、関連するタグが保持されていることを確認します。
GitLabへのプッシュ:

GitLabに bot-src-archive という名前の空のプロジェクトを作成します。
リモートリポジトリを追加します: git remote add gitlab <bot_src_archive_gitlab_url>。
ミラープッシュを実行します: git push --mirror gitlab。







D. タグ保持に関する注記: git filter-repo は履歴を書き換える際にタグを保持しようとします。タグは新しく生成された対応するコミットを指すように移動されます。分割フェーズ（ステップ C.2 および C.3）では、タグが指す元のコミットがフィルタリング後もそのサブプロジェクトの履歴に残っている場合（つまり、そのコミットが保持されたファイルを変更した場合）、そのタグはそのサブプロジェクトに保持されます。タグが指すコミットが、特定の分割で削除されたファイルのみを変更した場合、そのタグはその分割後のプロジェクトには現れません。これは意図した動作であり、各新プロジェクトにはその内容に関連するタグのみが含まれることを保証します。


E. Git LFS の検討: フェーズ1のクリーンアップ前の分析結果 (II.B) に基づき、apex-* ディレクトリや bot-src/bot-unittest ディレクトリに他のタイプの巨大ファイルがまだ存在する場合、対応するGitLabプロジェクト (apex-project または bot-src-archive) でGit LFSを設定・使用することを検討できます。bot-release が唯一の巨大ファイルの発生源であった場合は、LFSを導入する必要はありません。

IV. 移行後のリポジトリ管理 (apex-project 中心)

A. Gitブランチモデル:

現状: タグベースの方法は、構造化された開発プロセスが必要な apex-project には適していません。
推奨: GitHub Flow または Trunk-Based Development (TBD)。

理由: これらのモデルはシンプルで効率的であり、SQLコードの管理とイテレーションに適しています。GitHub Flowは機能開発を分離するためのフィーチャーブランチを提供し、マージリクエスト (MR) を通じて main にマージします。TBDはさらに簡潔で、短命なブランチまたは直接コミットに依存しますが、優れた自動化（CIは不要でもローカルでの自動テストスクリプトは有効）とフィーチャー管理が必要です。
選択基準: GitHub Flowの方が一般的に導入しやすいです。SQL変更管理に対するチームの規律が高く、変更頻度が高い場合はTBDも選択肢となります。



ブランチモデル概念図:
GitHub Flow:

main  ------------------------------------o---o (Merge MR)
       \                                 /
        feature/add-view ---------------o

Trunk-Based Development (TBD):

main  ---o---o---o---o---o---o---o---o---o---o
          \ /     \ /     \ /     \ /     \ /
           f1      f2      f3      f4      f5  (Short-lived feature/task branches or direct commits)



B. バージョン管理とリリースプロセス (apex-project):

ブランチ戦略:

GitHub Flow: main からフィーチャーブランチ (例: feature/add-customer-view) を作成し、完了後にMRを通じて main にマージします。main は安定し、デプロイ可能なSQLの状態を表します。
TBD: main が主幹です。開発は非常に短命なブランチ (例: task/update-order-proc) で行われ、迅速に main にマージされます。


タグ戦略: セマンティックバージョニング (例: v1.0.0, v1.1.0) を使用して、特定のデータベーススキーマバージョンを表す main ブランチ上のコミットにタグ付けします。タグは特定のスキーマバージョンを追跡し、デプロイするための重要なメカニズムです。



C. コードレビューワークフロー (apex-project):

main ブランチへのすべてのSQL変更は、GitLabマージリクエスト (MR) を通じて行われることを必須とします。
GitLabのレビュー機能（レビュアー割り当て、コメント、承認ルール、コードオーナー）を活用してSQLコードレビューを実施します。
レビューの重点項目：SQLロジックの正確性、パフォーマンスへの影響、命名規則、エラーハンドリング、プロジェクト標準への準拠など。



D. APEX SQLリポジトリ管理のベストプラクティス:

ファイル構成:

各データベースオブジェクト（テーブル、ビュー、パッケージ、プロシージャ、ファンクション、トリガーなど）を個別の .sql ファイルに保存します。
これらのファイルを整理するために、オブジェクトタイプ別 (tables/, views/, packages/) または機能モジュール別に明確なディレクトリ構造を使用します。


バージョン管理ツール連携:

SQL DeveloperなどのIDE統合機能も利用可能ですが、完全なGit機能と制御のためには、専用のGitクライアント（コマンドライン、Sourcetree、VS Code拡張機能など）の使用を推奨します。
SQLclの apex export コマンド（特に -split や -expType READABLE_YAML オプション）を使用してAPEXアプリケーションをファイルにエクスポートし、Git管理下に置きます。


変更管理:

データベース移行ツール（LiquibaseやFlywayなど。SQLclはLiquibaseを内蔵）を使用してスキーマ変更スクリプトを管理・適用します。これにより、変更が正しい順序で適用され、バージョン追跡が可能になります。
各論理的な変更に対して、明確で独立したコミットを作成します。コミットメッセージには変更の目的と内容を詳述します。


コード品質:

SQL Linter（例: SQLFluff）とFormatterを使用して、コードスタイルの一貫性を保ち、潜在的な問題を検出します。
コードレビュー（V.C節参照）を実施し、ロジック、パフォーマンス、セキュリティに重点を置きます。


環境管理: SQLスクリプト内に環境固有の接続文字列や設定をハードコーディングしないでください。変数や設定管理システムを使用して環境差を扱います。
APEX統合:

SQLclの apex export でエクスポートしたAPEXアプリケーションファイルをGitでバージョン管理します。
git diff を使用してAPEXアプリケーションコンポーネントの変更を確認します。
マージリクエストプロセスを通じてAPEXアプリケーションの変更をレビューします。



APEX SQL管理の概念図:
+-----------------------+      +---------------------+      +--------------------+


| APEX アプリケーション | ---->| SQLcl (apex export) | ---->| Git リポジトリ || (DB内) | | (YAML/SQL files) | | (apex-project) |+-----------------------+      +---------------------+      +--------------------+| ^| | git push/pullv |+--------------------+| 開発者のローカル環境 || (Gitクライアント) |+--------------------+| ^| | MR, レビューv |+--------------------+| GitLab || (コードレビュー) |+--------------------+|| 適用 (手動 or 移行ツール)v+--------------------+| 他のDB環境 || (テスト/本番) |+--------------------+```V. 詳細な移行実行計画
A. 移行前準備:

コミュニケーション: チームに移行計画、タイムスケジュール、コードフリーズ期間を明確に伝えます。2つの新しいリポジトリ (apex-project, bot-src-archive) の作成理由と目的、新しいリポジトリのURL、apex-project の新しいワークフローについて説明します。
バックアップ: Giteaリポジトリの最終的な完全バックアップを作成し、検証します。
GitLab設定: GitLabに apex-project と bot-src-archive の2つの空のプロジェクトを作成します。apex-project の main ブランチを保護ブランチとして設定します。ユーザー権限を設定します。

コードフリーズ: 計画された移行実行ウィンドウの開始前に、Giteaリポジトリへのコードコミットを停止します。

エラー対応: git filter-repo および git push --mirror の実行中にエラーが発生した場合は監視し、対応します。


C. 移行後検証リスト:

apex-project 検証:

ファイル構造が apex-apl/, apex-db/, apex-api/ のみを含むことを確認します。
履歴 (git log --stat, gitk --all) を確認し、Apex関連の変更のみが含まれることを確認します。
タグ (git tag -l) を確認し、Apex関連のすべての元のタグが保持されていることを確認します。
リポジトリサイズが大幅に減少したことを確認します。
main ブランチが保護され、権限設定が正しいことを確認します。


bot-src-archive 検証:

bot-src-archive をクローンします。
ファイル構造が bot-apl/bot-src/ と bot-unittest/ のみを含むことを確認します。
履歴を確認し、Bot関連の変更のみが含まれることを確認します。

D. 最終コミュニケーション: 移行の完了を宣言し、Giteaリポジトリのコードフリーズを解除します（または読み取り専用に設定します）。2つの新しいGitLabリポジトリのリンクと apex-project のワークフロー文書を提供します。サポート窓口を設けます。

B. ダウンタイム評価と対応:

主な影響はGiteaリポジトリのコードフリーズ期間です。ローカルでのクリーンアップと分割はフリーズ前に準備可能です。GitLabへのプッシュは既存のGiteaユーザーには影響しません。
フリーズ期間は数時間以内に抑えることを目指し、オフピーク時間帯（週末、夜間など）に実行します。ロールバック計画（Giteaバックアップを使用）を準備します。


C. CI/CD:

お客様のご要望に基づき、apex-project にはCI/CDを設定しません。
bot-src-archive はアーカイブ用リポジトリであり、通常CI/CDは不要です。


VII. GitLab機能の活用
A. apex-project:

コードレビュー: GitLabマージリクエストをSQLコードレビューに最大限活用します。
課題追跡: GitLab Issuesを使用してSQL関連のタスク、バグ、要件を追跡します。
プロジェクトボード: GitLab Boardsを使用してSQL開発タスクのワークフローを可視化します。
Wiki: apex-project 固有の開発標準、デプロイ手順などを記録します。


B. bot-src-archive:

アクセス制御: 厳格なアクセス権限を設定し、アーカイブとしての性質を保証します（例：ほとんどのユーザーは読み取り専用）。
コード閲覧: GitLab UIを使用して、過去のBotコードを簡単に閲覧・検索できます。


VIII. 核心的な提案のまとめ
分析先行: git filter-repo --analyze を使用して bot-release の影響を確認します。
移行戦略: ローカルクローン -> git filter-repo で bot-release をクリーンアップ -> クリーンアップ済みリポジトリをコピー -> git filter-repo でApex部分を分割 -> git filter-repo でBot部分を分割 -> それぞれを2つの新しいGitLabリポジトリにプッシュします。
ツール: 履歴のクリーンアップと分割には git filter-repo を使用します。
タグ: クリーンアップと分割プロセス中に、git filter-repo が関連するすべての元のタグを保持することを確認します。
apex-project 管理: GitHub FlowまたはTBDブランチモデルを採用し、MRベースのコードレビューを実施し、SQLバージョン管理のベストプラクティスに従います。
bot-src-archive 管理: 読み取り専用のアーカイブとして管理します。
コミュニケーション: 包括的なコミュニケーション計画を実行し、チームにリポジトリの分割と新しいプロセスを通知します。
CI/CD: apex-project にはCI/CDを設定しません。
GitLab活用: apex-project に対してはGitLabのコードレビュー、課題追跡などの機能を活用し、bot-src-archive に対してはアクセス制御を設定します。
