**プロジェクト概要**

  

本プロジェクトの目的は、Jira、Confluence、Mattermostを活用して業務を行う環境において、Jiraチケットのコメント欄で発生する@mention（メンション）の未返信を定期的に検知し、Confluence上の設定ページ（以下「設定表」）に定義された監視ルールに基づいて該当ユーザーやMattermostチャンネルへ通知を送信する自動化ツールを構築することです。

  

主な機能は以下のとおりです：

  

1.  Confluence上の設定表を読み込み、チェック対象のルールを取得する。

2. ルールに従い、定期的にJiraのJQLを実行し、各設定ルールに基づく検知を行う。

3. 条件に合致するチケットがあれば解析し、必要に応じて通知を送信する。

4. 通知内容を生成し、Mattermost APIを使用して専用チャンネルまたは対象ユーザーに送信する。

5. マルチスレッド（多スレッド）で処理を行い、パフォーマンスを向上する。

6. 実行中にエラーが発生した場合、Confluenceの設定表の該当行にエラーログを書き戻す機能を提供する。

7. エラーの書き戻しの際は、Confluenceのページバージョンが更新されていないかを確認し、更新されている場合は再取得後に反映し、競合を回避する。

  

**目的****および****機能概要**

  

1.  Jiraの@mention未返信の定期検知

• 「自分が他者に送信したコメントに、相手から返信がない状況」をチェックするかどうか、および該当キーワード（keyword）を判断する機能を提供します。

• 対象プロジェクトや、クローズ済みチケット（ClosedやDoneなどのステータス）の扱いなど、柔軟にフィルタリングが可能です。

2. 通知方法のカスタマイズ

• 「漏れがなかった場合も報告を受け取りたい」などの要望に応じて、空通知の有無を設定可能です。

• 通知先を特定ユーザー（MattermostユーザーIDなど）またはチャンネルに指定できます。

3. マルチスレッド実装

• 設定表のエントリ数が100件以上のケースを想定し、指定したスレッド数（例：10スレッドなど）で同時にJira検索や通知送信を実施します。

• 処理効率を向上させ、大規模なチケット監視もスムーズに行います。

4.  Jira検索（ページング対応）

•  Jira REST API（/api/2/search）では、1回のリクエストで最大50件のIssueしか取得できないため、startAtやmaxResultsのパラメータを使用したページング処理を実装し、すべての対象チケットを連続取得します。

5.  エラーログのConfluence回写

• ルール実行中に発生したエラー（Jira接続エラー、Mattermost送信エラーなど）はメモリ上や一時領域に保持·収集し、最終的にConfluenceの設定表の該当行にまとめて出力します。

• 回写時にページバージョンが更新されている場合は、再取得·内容マージのうえ更新を行い、競合を防ぎます。

  

**システム****全体****のフロー**

  

1. タスクのトリガー

•  cronやQuartz等のスケジューラによって日次もしくは所定の時間に自動実行されます。

• テストなどのために手動で起動することも可能です。

2.  Confluence設定の取得

•  Confluence API（例：/rest/api/content/{pageId}）を使用し、設定ページの内容を取得します。

• 表形式のデータをパースし、以下の情報を各行から読み取ります：

• 通知対象ユーザー（MattermostユーザーIDまたはメールアドレス  等）

• 通知スケジュール（cron式など）

• 自分から他者へのコメントに返信がないかチェックするかどうか（boolean）

•  キーワード（keyword）

• 未返信がない場合にも通知を行うかどうか（boolean）

• クローズチケットも対象とするかどうか（boolean）

•  対象プロジェクト（Project Key複数可）

• エラーログ格納先（エラー内容の書き込み欄）

3. マルチスレッドでの実行

• 取得した設定の各行をタスクとしてキューに格納します。

• 設定で指定されたスレッド数（例：10スレッド）をもつスレッドプールを用意し、各行ごとにスレッドを割り当て並列実行します。

4.  Jira検索およびチェック（各設定行ごと）

• 設定内容に基づき、JQLを組み立てます。

•  例：·project = XXX·status not in (Closed, Done) / status in (Closed)（クローズチケットを含むかどうか）·コメント内に特定キーワードが含まれるかどうか 等

•  /api/2/searchを使い、maxResults=50を上限とするページングを実施しながらIssueを全件取得します。

• 取得したIssueの各コメントを確認し、以下のような条件をチェックします：

• 特定ユーザーへの@mentionがあり、返信がなかったかどうか。

• 「自分が投稿したコメントに対して相手から返信がないか」をチェックする場合は、コメントの投稿者と@mention対象の関係、および後続コメントの有無を追加でチェックします。

5. 通知の生成と送信

• 上記の条件に合致した場合、または「未返信がない場合でも報告」がtrueの場合、通知メッセージを生成します。

• 対象Issueの情報、コメントURL、未返信状況など必要な情報をまとめます。

•  Mattermost API（例：/api/v4/posts）を用いて、指定されたユーザーまたはチャンネルにMarkdown形式などでメッセージを投稿します。

6.  エラーの収集とConfluenceへの書き戻し

• 実行中に発生したエラーを取りまとめます（Jira接続障害、Mattermost送信失敗などのスタックトレース）。

• すべてのタスクが終了後、Confluence設定ページの最新版バージョンを取得します：

• もし前回取得時のバージョンと異なる場合は、再取得により内容をマージのうえでエラー内容を該当行に書き込みます。

•  Confluence API（例：/rest/api/content/{pageId}）でページを更新し、エラーログの書き込みを行います。

  

**インタフェースおよび****技術仕様**

  

1.  Jira REST API

•  エンドポイント： /rest/api/2/search

•  認証：Basic認証（ユーザー名＋トークン）やOAuth、Cookie認証などを利用可能です。

•  ページング：·startAtとmaxResultsを用いて複数回に分けて最大件数超過分を取得します。

• 同時アクセス可能件数などのレート制限にも留意し、必要に応じてリトライやキューを実装してください。

2.  Confluence REST API

•  エンドポイント： /rest/api/content/{pageId}

•  認証：Jira同様にBasic認証＋トークンやCookie認証を想定。

•  ページ内容：storage形式（application/jsonまたはtext/xml）を含んでおり、表をHTML/XHTML形式で確認·編集します。

• バージョン管理：更新時はversion指定が必須であり、衝突が発生する場合は差し戻し対応（再取得と更新）を行う必要があります。

3.  Mattermost API

•  エンドポイント： /api/v4/posts

•  認証：トークン認証（Bearer Token）などを使用。

• メッセージ送信時にMarkdownなどリッチテキストでの投稿が可能です。

4. マルチスレッド実装

• 固定サイズまたは設定可能なスレッドプールを利用します。

• 各設定行に1つの「チェックと通知」タスクを割り当てます。

• さらに状況に応じて細分化が必要な場合は追加の並列化を考慮します。

5. ログおよび例外処理

• システムログ：重要な処理ステップと結果を記録し、トラブルシュートを容易にします。

• エラーログ：各設定項目ごとに紐付けし、最終的にConfluenceにまとめて書き込みます。クラス名やスタックトレースも保持してください。

6. スケジューリング（定期実行）

•  Quartz、Spring Scheduler、LinuxのCrontabなどを利用して周期的に実行可能です。

•  Confluenceの設定表で指定されたcron式を解析し、必要なタイミングで実行する設計も可能です。

  

  **設定例（Confluence表イメージ）**

• 「通知時刻」はcron式などで指定し、解析後に通知スケジュールを設定。

• 「クローズチケットをチェックする」はtrueの場合ClosedやDone状態を含むJiraチケットを対象に含める。falseの場合はそれらを除外。

•  キーワードkeywordは@mention以外にもcomment内に含まれる特定フレーズなどを指定する場合にも使用。

• 「エラー内容」はツール実行時のエラーを記録する列で、手動調査にも活用します。
**デプロイおよび****実行方法**

  

1.  事前準備

•  Jira、Confluence、Mattermostへのアクセスに必要なTokenまたはユーザー名/パスワードを取得し、設定ファイルに正しく記載してください。

• 本ツールがこれらのシステムAPIにアクセスできる権限を確認してください。

• （オプション）キャッシュや増分管理が必要な場合はデータベースやローカルファイル領域を考慮してください。

2. 動作環境依存関係

•  Javaの場合はJDK 1.8以上、あるいは利用する言語に応じたランタイム環境。

•  Maven / Gradle / npm等のパッケージ管理ツール。

•  HTTPクライアントやJSON/XMLパーサ、多スレッドスケジューラなどのライブラリ。

3.  設定ファイル例

•  application.properties / config.ymlなど：

4.  実行方法

•  Java版の場合：ビルド後に java -jar xxx.jar を実行。

•  IDE（IntelliJ / Eclipseなど）からmainメソッドを直接起動して動作確認することも可能。

• ログはコンソール出力またはログファイルへ出力可能です。

  

**開発上****の****注意点****と****拡張性**

  

1.  Confluence設定表の柔軟性

• 列やタイトルが変更された場合、パース用ロジックの修正が必要になる可能性があります。

• 設定表にユニークIDを付与しておくと行の同期やマージ処理が容易です。

2. セキュリティ·安定性

•  HTTPS通信を利用し、機密情報（トークン等）を安全に取り扱ってください。

• 同時にJiraやMattermostに大量アクセスが発生しないよう、レートリミットや再試行ロジックを検討してください。

3.  リトライ機構

• ネットワーク障害やAPI不調での一時的な失敗に備え、タスク失敗時には数回リトライする設計が推奨されます。

4. 多言語対応

• 将来的に多言語ユーザーが混在する場合は、Mattermost通知のテンプレートを多言語化することを検討してください。

5.  ログ追跡

• 実行ごとにトレースIDなどを付与し、一連のログを相互に紐付けられるようにすると保守がしやすくなります。

6.  機能拡張

•  SlackやMicrosoft Teams等、他の連携先への通知も追加実装可能です。

• 通知文面テンプレートを設定表で管理し、柔軟にカスタマイズできる機能も追加検討すると良いでしょう。
