# 全面排查服务器中 API Token 的方法

**问题背景：** 当前有若干 VM 服务器上运行着以服务或 Docker 形式部署的服务，这些服务使用了与生产环境相同的 API Token。需要在特定机器范围内，手动、高效地搜索并整理出所有相关 Token 信息，包括本地配置文件、环境变量以及 Git 仓库中的敏感信息。基于给定条件，我们可以制定一套系统可靠的排查方法。

## 方法概述

要全面排查这些 API Token，我们应采取多管齐下的策略，主要涵盖以下几个方面：

* **扫描本地配置文件：** 搜索各类配置文件（如 YAML、JSON、INI、Toml、XML、Shell 脚本等）中是否包含 Token 字符串。
* **检查环境变量：** 查看服务器和容器的环境变量设置，寻找任何包含 Token 的项。
* **扫描 Git 仓库：** 检查在服务器上克隆的 Git 仓库（及其历史提交）中是否存在 Token 泄露。

在执行上述步骤时，我们将利用关键词和 Token 特征进行搜索。例如，Token 通常是一串字符，有时带有固定前缀，因此可以利用已知**前缀**或常见敏感关键词（如“token”、“secret”、“password”等）来定位。

接下来，我们分别详细说明各步骤的实施方法。

## 扫描本地配置文件中的 Token

首先，需要遍历服务器上的配置文件，查找其中是否包含 API Token 或其他敏感凭据。考虑到配置文件可能采用多种格式，我们应关注常见的配置文件类型及扩展名，包括：

* **YAML/JSON 格式：** `.yaml`, `.yml`, `.json` 等，用于应用配置、凭据管理等。
* **INI/Conf 格式：** `.ini`, `.conf`, `.cfg`, `.config` 等，用于软件设置或服务配置。
* **脚本及其它：** Shell 脚本（`.sh`）、环境文件（如 `.env`）、XML 文件（`.xml`）等等。

**扫描方法：** 在 Linux 系统（Amazon Linux 或 Oracle Linux）上，可以使用诸如 `grep` 或 `find` 等命令进行递归搜索。具体策略包括：

* **按关键词搜索：** 利用 `grep -R` 在目录下递归搜索，例如：

  ```bash
  grep -R -n -i "token" /path/to/search
  ```

  该命令会搜索指定路径下所有文件内容，查找字符串“token”（不区分大小写），并输出匹配的文件名和行号。我们也可以使用其他关键词组合，如“password”、“secret”、“key”等，以确保全面覆盖。这种基于关键词的搜索非常高效，安全研究人员常用类似方法来定位配置中的凭据。例如，有报告指出，攻击者会搜索诸如“password”等关键词来快速定位包含凭据或API密钥的文件。

* **按文件类型过滤：** 为提高准确性，可结合 `find` 命令按文件扩展名过滤后再用 `grep`。例如：

  ```bash
  find / -type f \( -name "*.yml" -o -name "*.json" -o -name "*.conf" -o -name "*.ini" -o -name "*.env" \) -exec grep -H "TOKEN_PREFIX" {} \;
  ```

  这里我们假设 Token 有已知的固定前缀“TOKEN\_PREFIX”，上述命令将搜索系统中所有符合这些扩展名的文件，查找包含该前缀的行。使用固定前缀能显著减少误报，并快速定位目标字符串。许多 API 密钥都有固定格式或前缀，例如 AWS 密钥等，其标准格式能够帮助我们更容易地识别它们。另外，也可以利用正则表达式进一步精确匹配 Token 模式（如长度、字符集等），以防止错漏。

* **注意隐藏文件：** 别忘了检查隐藏配置文件，比如以“.”开头的文件（如 `.env`、`.bash_profile` 等）。特别是 `.env` 文件通常用于存放应用的环境配置，其中可能包含敏感信息（如 API Token、数据库密码）。攻击者往往通过列出隐藏文件发现其中的凭据，因此我们在排查时也需要包括这些文件。

在执行上述搜索时，应以特定目标服务的安装目录或常用配置路径为重点（例如 `/etc/`、应用部署目录等），以减少干扰信息。同时确保有足够权限读取这些文件，否则需要使用 `sudo` 提权。逐一记录发现Token的文件路径、配置项名称和值，以便后续汇总。

## 检查环境变量中的 Token

除了文件，很多服务将 Token 配置在环境变量中，比如通过系统启动脚本、Docker Compose、Kubernetes Secrets等方式注入。根据条件，我们可以访问VM的运行环境，因此需要检查：

* **系统环境变量：** 登录服务器后，使用 `printenv` 或 `env` 查看全局环境变量列表。结合之前的关键词，用 `grep` 过滤，如：

  ```bash
  env | egrep -i "token|secret|key|password"
  ```

  这样可以快速找出变量名或值中包含这些关键词的环境变量。如果 Token 带有固定前缀，也可以直接 `env | grep TOKEN_PREFIX` 搜索。

* **服务启动配置：** 针对通过 Systemd 启动的服务，查看对应的 service 文件（通常位于 `/etc/systemd/system/` 或 `/lib/systemd/system/`）以及 `/etc/sysconfig/` 或 `/etc/default/` 下的环境配置文件，寻找其中的环境变量设置。例如 systemd 单元文件里 `Environment=` 或 `EnvironmentFile=` 指定的内容。使用 `grep "TOKEN_PREFIX" /etc/systemd/system/yourservice.service` 等命令定位其中是否有 Token。【**注:** Amazon Linux/Oracle Linux 基于 RHEL/CentOS，服务配置通常遵循相似路径】

* **Docker 容器环境：** 如果服务以 Docker 形式运行，需要检查容器内或部署配置中的环境变量：

  * 查看部署配置（如 Docker Compose 文件 `.yml`、运行脚本）中是否有 token 提供。用关键词搜索这些文件即可（方法同上）。
  * 检查正在运行的容器环境：可以使用 `docker inspect <container>` 来输出容器配置信息，包括环境变量。之后同样用 `grep` 搜索其中的 Token 前缀或关键词。**提示：** Docker 的镜像构建文件（Dockerfile）里设置的环境变量，或运行时 `-e` 传入的变量，都能通过此方式查出。正如安全报告所示，容器的描述信息中往往能看到设置的环境变量，其中可能包含敏感的Token。攻击者团伙 TeamTNT 就曾专门扫描入侵容器的环境变量以窃取云服务凭证。

将找到的环境变量名和值记录下来，并标明所属的服务/容器。特别注意，一些 Token 可能存储在内存中而非文件，这种情况下只能通过环境变量或运行时配置来获取。

## 扫描 Git 仓库中的敏感信息

最后，需要检查托管于 Gitea 平台的 Git 仓库（在各目标机器上已 clone 的本地仓库）中是否存在 Token 泄露。一方面，Token 可能直接保存在仓库的配置文件中；另一方面，即使未在当前版本中，历史提交中也可能有过泄露记录。

**扫描工作区文件：** 直接在仓库目录下使用与上述类似的 grep 方法，搜索包含 Token 前缀或相关关键词的文件内容。重点检查典型文件：

* 配置文件（同上文提及的各种格式）。
* 代码文件中可能硬编码的 Token（如 `.py`, `.js`, `.java` 等代码中出现的字符串）。
* README 或文档中意外包含的凭据。

**扫描 Git 历史：** 为了发现**历史提交**中是否曾泄露 Token，可使用专门工具或 git 命令：

* 利用扫描工具：例如 **Gitleaks**、**TruffleHog**、**GGShield** 等开源工具，可以自动扫描整个 Git 历史中的敏感信息。这些工具会搜索常见密钥模式（包括自定义前缀）并标记高风险内容，适合全面检查。使用它们可以快速定位过去提交中是否有 API Token 等泄露记录。
* 使用 Git 自带命令：也可以手动运行 `git log -p` 或 `git grep` 在历史提交中查找。例如：

  ```bash
  git grep -I -n "TOKEN_PREFIX" $(git rev-list --all)
  ```

  这将遍历所有提交，查找包含特定前缀的内容。需要注意此过程较慢但无额外依赖。

若发现仓库中存在 Token（无论当前还是历史），应进一步确认是否与生产 Token 相符，并检查访问权限。务必将发现的位置（文件路径和所在 commit哈希）记录下来，以评估泄露范围。

## 汇总结果与下一步措施

按照上述方法，对指定的每台目标 VM 进行逐项检查，我们将得到以下信息汇总：

* 在**本地配置文件**中发现的 Token：列出文件路径、相关配置键名和值。如果值即为生产 Token，应予以标记。
* 在**环境变量**中发现的 Token：列出变量名、所在服务/容器及其值（或标记为与生产一致）。
* 在**Git 仓库**中发现的 Token：说明是在当前文件还是历史提交，给出文件路径或 commit 引用。

通过整理这些信息，可以全面了解哪些服务在使用生产环境的 API Token，以及这些 Token 存储在哪里。这有助于我们评估安全风险并采取后续措施（例如更换 Token、避免在不必要的环境中使用生产凭证）。

**附加建议：** 在排查过程中，如发现其他敏感信息（密码、密钥等），也应一并记录。所列的关键词搜索方法同样适用于发现诸如密码或私钥等其他秘密。另外，确保在排查完成后，针对此次发现制定凭据更换和管理策略，减少同一Token多处使用的情况，以提升安全性。

通过以上步骤，我们建立了一套可靠而高效的手动检查方法，能够较全面地覆盖服务器文件系统、运行环境和代码仓库，尽可能找到所有涉及生产 API Token 的痕迹，为后续安全加固提供依据。
