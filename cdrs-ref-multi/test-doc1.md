以下では、先にまとめた**全体運用テスト（総合テスト）**の計画やテスト観点に加え、新たに頂いた要件を反映し、補足・更新した内容を提示します。今回追加になった要件は以下のとおりです。

1. **サービス修正リリース後の最低限実施する確認**  
   1.1 タスク一括出力（正常ケース／エラーケース）  
   1.2 通知設定（定期実行・即時実行で通知が正常／エラーともに行われるか）  
   1.3 他の8機能についても最低限の確認  
2. **ログローテーション、ログ出力レベル網羅確認**  
3. **追いつき開発（追従開発）との整合性確認テスト**  
4. **CI/CD パイプライン実現の確認**  
5. **現行機能保障の確認（マニュアル外の仕様も含め、リファ後も動作が保証されているか）**

以下に、既存のテスト計画へこれらをどのように組み込むかを示します。

## 1. 追加要件の整理と全体テスト計画への統合

### 1.1 サービス修正リリース直後の最低限確認

- **目的**: リリース後すぐに大きな問題（致命的な障害や主要機能の不具合）がないかを短時間で確認し、本稼働に影響が出ないようにする。
- **主なチェックポイント**:  
  1. **タスク一括出力**  
     - **正常ケース**: 通常のタスク一括処理が想定どおりに実行され、正しい結果が取得できるか。  
     - **エラーケース**: 入力データ不備・ネットワーク障害などによりタスク一括処理が一部失敗した場合に、エラー情報やログが正しく出力・通知されるか。  
  2. **通知設定**  
     - **定期実行タスク**: スケジュール時刻到来後の通知（メール／Mattermost）が行われるか、成功／失敗ともに正しく通知されるか。  
     - **即時実行タスク**: ユーザー操作によるタスク実行の通知が正常に行われるか、成功／失敗時のメッセージが正しいか。  
  3. **他の8機能の最低限確認**  
     - システム独自に定義された8つの主要機能（例：JIRAチケット管理機能、Confluenceページ管理機能、など）が正常動作することを、軽い動作確認で担保する。  
     - 例：主要APIコールがエラーなく応答し、データベース更新・UI操作や連携先への書き込みが正常に行われるかなど。

- **手法**: “スモークテスト”に近い簡易的なチェックリストを用意し、リリース後に即座に実行する。問題があれば即時ロールバックや追加対応を検討する。

### 1.2 ログローテーション、ログ出力レベル網羅確認

- **目的**: 本番運用でログファイルが肥大化しないようにする仕組み（ログローテーション）や、適切なログレベル（DEBUG / INFO / WARNING / ERROR / CRITICAL など）が実際に切り替え可能であり、出力内容が想定どおりかを確認する。
- **主なチェックポイント**:
  - ログローテーションが設定どおりのタイミング（サイズ・日付ベースなど）で実行されるか。  
  - ログ出力レベルを切り替えた際に、不要な詳細情報や機密情報が含まれないか、逆に必要な情報が欠落しないか。  
  - ローテーション後の古いログの扱い（圧縮／削除など）が正常に行われるか。  
  - 実運用中に想定される期間を超えてログが溜まった場合の動作（ディスク容量制限など）。

### 1.3 追いつき開発との整合性確認テスト

- **追いつき開発（追従開発）**: 並行で進む他プロジェクトや追加機能開発の成果物を、本システムに組み込みながら開発を進める場合、その進捗やバージョン差異に追従する形で開発を行うこと。
- **目的**: リリース時点で他チームの変更や追加機能と矛盾・競合が起きていないかを確認する。
- **主なチェックポイント**:
  - 外部APIのバージョンアップや仕様変更に伴うテスト（JIRA / Confluence / APEX API の更新など）。  
  - システム間インターフェース（JSONスキーマ、パラメータ追加・廃止）に変更があった場合、本システムが正しく対応しているか。  
  - CIパイプライン上で他プロジェクトのモジュールを最新化した場合に、テストがすべてグリーンになるか。

### 1.4 CI/CDパイプライン実現の確認

- **目的**: 開発～テスト～デプロイを自動化するCI/CDパイプラインの構築が完了しているか、およびそのパイプラインを用いたビルド・テスト・デプロイがエラーなく進むかを確認する。
- **主なチェックポイント**:
  - 各ステージ（ビルド、ユニットテスト、統合テスト、デプロイなど）のジョブが成功するか。  
  - テスト結果レポート（例えばJUnit形式やカバレッジレポートなど）がパイプライン上に適切に出力されるか。  
  - デプロイ先（ステージング／本番）への自動反映が正しく行われるか、ロールバック手順が用意されているか。  
  - パイプライン上での通知（Slack、Mattermost、メールなど）が正しく行われるか。

### 1.5 現行機能保障の確認（マニュアル外仕様のリファ後の確認）

- **目的**: 過去バージョンから暗黙的に運用していた機能（マニュアルに明文化されていないが、利用ユーザーが依存している仕様）もリリース後に問題なく動作し続けているかをチェック。
- **主なチェックポイント**:
  - マニュアルに載っていないが、既存運用で重要な機能や画面の挙動が変化していないか。  
  - 既存のスクリプトやバッチが呼び出していた内部APIのパラメータやレスポンスが破壊的変更を受けていないか。  
  - ユーザー（運用担当）が行う手順、慣習的な運用方法に差異がないか（例：ある裏コマンドラインオプションを常用していた等）。  
  - 過去リファレンスや既存ユーザーのヒアリングを行い、抜け漏れのないように確認する。

## 3. 各テスト項目への補足

1. **サービス修正リリース後の最低限確認（スモークテスト）**  
   - 実際の運用担当が使う想定の画面・コマンド・APIを短時間で確認する“チェックリスト”を事前に整備しておく。  
   - リリース直後に確認し、問題があればすぐにロールバック可能な体制（運用窓口、開発チームへの連絡ルートなど）を準備しておく。

2. **ログローテーション・ログ出力レベル**  
   - ローテーション時のファイル名（例：`app.log`→`app.log.1`など）や時刻フォーマット、圧縮の有無などをテストし、異常がないかを確認。  
   - ログレベル切り替えにより、デバッグ情報が増えた際にリソース（ディスク容量、NW帯域）へ影響がないかも留意。

3. **追いつき開発の整合性確認**  
   - 他チームのリリース計画・バージョン管理表などを参照し、本システム側でも合致するタイミングで総合テストを実施。  
   - API変更点のドキュメントを照らし合わせ、モックや実機テスト環境で合わせて検証する。

4. **CI/CDパイプライン**  
   - 単に動けばよいだけでなく、テストの失敗時や途中キャンセル時に、ステージングや本番環境への影響がないように設計する。  
   - 結果通知（成功／失敗）を関連者（開発・運用）へ送る機能も含めて確認。

5. **現行機能保障（マニュアル外仕様）**  
   - ユーザーや運用担当とのヒアリングをあらためて実施し、暗黙的に利用されている仕様を洗い出し、テストケースに反映。  
   - 将来的にはマニュアル外仕様を正式にドキュメント化し、バージョン管理の一部として扱うことが望ましい。

## 4. まとめ

- 今回追加された要件は、**リリース直後の短期的な検証（スモークテスト）**、**ログローテーションやログレベルの運用面**、**他チームの開発同期**、**CI/CDの動作確認**、そして**現在運用中の機能を壊さない保証**といった、より**実務・運用に即した観点**が中心となっています。  
- これらを既存の「全体運用テスト」の計画やテストマトリクスに統合することで、漏れなくかつ効率的に総合テストを実施できます。  
- 特にリリース後の“即時チェック”や“現行仕様の暗黙的依存”などは、チームメンバー間や運用担当との情報共有・認識合わせが必須です。  
- 引き続き、テスト計画書・チェックリストを明確化し、実運用に合わせてアップデートを行ってください。

### 参考ポイント

1. **必要に応じてシナリオをさらに細分化**  
   - 例えばF1の「大量タスク & 高負荷シナリオ」を、同時100件・同時1000件・深夜バッチなど具体的に条件を変え、複数ケースを用意。  
   - B1ログローテーションも「ファイルサイズ超過」「日付ベース」「1日1回のバッチローテーション」など細かく分ける。

2. **「○」「×」「-」「NA」等で結果を残す**  
   - 実際にテストを実行する際は、各セルに「○」(成功)、「×」(失敗)、「-」(該当なし)などを記録。  
   - 失敗の場合は、別途不具合チケットを発行し、修正後に再テストを実施する運用を行う。

3. **シナリオごとのスクリプトやチェックリストを作成**  
   - 行(シナリオ)が大項目になるので、実際には詳細な手順書（How to 実行するか）と期待結果(○○が返ってくる)をまとめるとより分かりやすい。  
   - リリース直後チェック（A1～A4）は、誰が何分以内にどこを確認するか明確にする。

4. **チーム内で定期的に更新**  
   - 新機能追加や仕様変更が入ったら、このマトリクスも随時アップデートしていく。  
   - 特に「追いつき開発」や「CI/CDパイプライン」は動的に変わりやすいので注意。
